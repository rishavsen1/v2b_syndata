02

Resolution69 typed knobs, one priority funnel, full provenance

Descriptors expand into knob values; four sources compete for each knob and the strongest wins. Every resolved value remembers its origin — the manifest can answer “why is this number what it is?” for all 69.

1 · CLI --overridesource: explicit
2 · scenario YAML overridessource: explicit
3 · descriptor expansionsource: descriptor:* / calibration:* / hand_specified:*
4 · knobs.yaml defaultsource: default
Two channels, one namespace. Registry knobs (ev_fleet.ev_count…) are typed and range-checked from knobs.yaml; deep-channel leaves (user_behavior.region_distributions.<region>.<dist>.<param>) carry the calibrated fits and are validated against DIST_PARAM_RANGES — an out-of-range override is rejected, not clamped.

How each knob gets its value →

every knob → first hit wins: CLI override → scenario overrides → descriptor expansion → knobs.yaml default
value → type + range check (registry) or DIST_PARAM_RANGES check (deep) → KnobValue(value, source)
manifest.knob_resolution → all 69+ resolved values with their source labels (the audit trail)
03

SeedingOne seed, hash-keyed into hundreds of independent streams

Every random draw belongs to a named stream: SeedSequence(seed, spawn_key=SHA-256(node_name)[, car_id]). Keys are content-hashes, not spawn order — adding a new node can never shift another node's randomness.

seed = 42root entropy
→
SHA-256("A_user")per-car stream, car 0…N
·
SHA-256("sessions:2024-01-03")per car, per day
·
SHA-256("dirichlet:axes")isolated aux stream
·
SHA-256("weather_realization")per-sample weather draw
The contract this buys: (scenario, seed) → byte-identical CSVs, forever. Guarded by test_reproducibility.py + test_determinism_stress.py; Python's salted hash() is banned.

How a random number is generated →

node stream → SHA-256(node name) as spawn key → SeedSequence(seed, key) → default_rng
per-car stream → spawn key = (SHA-256(node), car_id) — car 7's draws never move car 8's
per-day stream → node name embeds the date: "sessions:2024-01-03" — each day independent

EnergyPlus15-min timestep · 8 meters
→
Joules → kWenergy-per-tick ÷ 900 s ÷ 1000
→
building_load.csvpeak scaling (optional)
  • Why ÷ 900 000? EnergyPlus meters report energy in Joules per 15-minute tick. Average power over the tick = energy ÷ time: J ÷ 900 s gives watts, ÷ 1000 more gives kilowatts — so kW = J / (900 × 1000) = J / 900 000. e.g. 90 MJ in one tick → 90 000 000 / 900 000 = 100 kW.
  • flex = Cooling + Heating + Fans + WaterSystems — the schedulable half.
  • inflex = Interior/Exterior Lights + Interior/Exterior Equipment — the must-serve half.
  • mixed archetype = office and retail prototypes run separately, averaged 50/50.
  • peak_kw_scaling: true (default) pins max(power) to the descriptor's peak_kw exactly; false lets weather drive the peak.

How each feature is generated →

weather frame → EPW(tmyx_station) → +ΔT dry-bulb · ×solar · +Δdew (RH recomputed) · ×wind
power_flex_kw[t] → E+ meters Cooling+Heating+Fans+Water → J per tick ÷ 900 s ÷ 1000
power_inflex_kw[t] → E+ meters Int/Ext Lights + Int/Ext Equipment → same conversion
power_kw[t] → flex + inflex → if peak_kw_scaling: × (peak_kw / max) so max == peak_kw
Same weather everywhere. The one perturbed weather frame feeds EnergyPlus, the PV model, and the exported weather_data.csv — load, PV and weather can never disagree. That's what makes the input-side weather layer physically faithful.
06

SessionsPer car, per day: draw from fitted distributions, then survive five gates

Each car appears on a weekday with probability φ. On appearance, one joint draw — arrival and dwell coupled through a Gaussian copula — then the attempt must pass every gate or be redrawn (≤ 5 tries, else the car-day is dropped). Enforcement is rejection, never clamping.

Arrival hour — density

■ synthetic default: TruncNorm(μ 8.5, σ 2(1−κ)), κ=0.85, window [6, 20]  ·  ■ ACN-calibrated rare_consistent: 2-component mixture w₁ .42·N(8.21, 1.25) + .58·N(14.54, 3.52), window [4, 22] — the real workplace data is bimodal (morning commute + after-lunch).

Dwell hours — density

■ synthetic: Weibull(k 2.0, λ 8(0.5+φ)), φ=0.9, clip [0.5, 14] h  ·  ■ ACN rare_consistent: Weibull(k 1.17, λ 5.63) — heavier short-stay mass than the synthetic prior.

Arrival SoC — density

■ synthetic prior: Beta(4, 6) − 0.003·δ shift (drawn at δ=60 km)  ·  ■ ACN fit: Beta(3.70, 5.59). Clipped to each car's [min, max] SoC band. (SoC is a modeled prior — see Act 11.)

One car-day on the clock

A session lives inside one calendar day (C12), on the 15-min grid, non-overlapping with the car's previous session, and its energy demand must fit the dwell × charger-rate budget (D5).

How each session field is generated →

appears today? → Bernoulli(φ) on weekdays (weekends: φ × weekend_factor, off by default)
(u_arr, u_dwell) → one Gaussian-copula pair with the region's ρ (ρ≈0 → independent uniforms)
arrival → TruncNorm⁻¹(u_arr | μ, σ, window) → snapped to the 15-min grid
dwell / duration_sec → Weibull⁻¹(u_dwell | k, λ) → clipped [0.5, 14] h → departure = arrival + dwell
arrival_soc → Beta(α, β) − 0.003·δ → clipped to the car's [min, max] band
required_soc_at_depart → calibrated Beta (or TruncNorm fallback) → floor max(min_depart, arrival+0.01), cap max_allowed
session row → keep only if all 5 gates pass → else redraw (≤ 5 tries) → else drop the car-day

The five gates (atomic — fail any ⇒ redraw everything)

C12 · same daydeparture date == arrival date; no overnight stays.
boundsarrival ≥ window start, departure ≤ window end.
non-overlaparrival ≥ this car's previous departure.
D6 · directionrequired SoC > arrival SoC floor = max(min_depart, arr + 0.01)
D5 · reachability(req−arr)/100 · cap_kWh ≤ rate_kW · dwell_h

Pure physics, zero RNG. pv_type: none (the default) short-circuits to an all-zero series with no weather I/O — which is how PV was added without breaking the bitwise contract.

TOU price day (defaults)

$0.137 off-peak / $0.178 peak inside hours [6, 22) — deterministic; the only price randomness is the optional adversarial jitter (Act 8).

DR event intensity λ(T, t) — shape

Thinned inhomogeneous Poisson: λ_base 0.05 ev/h × season gate × weekday 3:1 × temperature ramp (0.1 below 80 °F → 3.0 above 100 °F) × time-of-day (peak 15–17 h, zero outside 12–20 h), capped per program. Notification: CBP 24 h, BIP/ELRP 2 h.

Stationary battery — specs only, deliberately

One row: capacity / power / η / SoC window / initial SoC (preset or explicit). No dispatch schedule is generated — dispatch is a decision for the downstream optimizer, not exogenous data.

How each feature is generated →

price_per_kwh[t] → peak $ if hour ∈ peak_window else off-peak $ (flat tariff: one price everywhere)

08

Two kinds of randomnessInput-side weather realization vs output-side measurement noise

Weather layer (input-side)

Each sample draws its own ΔT / solar× / Δdew / wind× (profiles: slight σ=1 °C · moderate 2.5 °C · strong 5 °C), then EnergyPlus re-simulates. The whole load curve moves coherently — physically faithful variance, never violates a constraint.

Noise layer (output-side)

Gaussian jitter applied to the finished CSVs (profiles: clean 0 → adversarial 15% load / 45 min arrivals / 20% SoC). Guard rails re-clamp C4/D6 and truncate D5 (headroom 1.04) — but the validator's strict envelope (1.0) can still flag jittered sessions.
Field-proven: for Monte-Carlo batches use noise: clean + a weather profile. Diversity comes from the weather re-sim + per-sample seeds; validation stays at 0 hard errors (verified on 18,000 units). tmyx_stochastic-style output jitter flags ~50% of units on strict D5.

How each perturbation is generated →

weather sample → ΔT ~ N(0, σ_T) · solar× ~ N(1, σ_s) · Δdew ~ N(0, σ_d) · wind× ~ N(1, σ_w) → full EnergyPlus re-run
noisy load[t] → load[t] × (1 + N(0, jitter_pct)) → clipped ≥ 0
noisy arrival → arrival + N(0, jitter_min) → re-snapped to grid, bounded so ≥ 30 min dwell + same day survive (C4/C12)
noisy arrival_soc → soc + N(0, jitter) → clamped to [car min, required − 0.1] (D6 survives)
D5 repair → truncate required_soc to arrival + rate·dwell·1.04 → drop if unrepairable → counts logged to manifest
dr dropout → Bernoulli(p) per event · noisy price[t] → price × (1 + N(0, σ)) (adversarial only)
09

The gates at the exitE5 feasibility + nine invariant families, then the manifest seals it

Before anything ships, the run is checked against the full invariant wall — hard families raise, soft ones warn:

A · schemaB · referential idsC · temporal (C1–C12) D · SoC physics (D1–D7)E · charger conventions G · behavioral axesH · tariff / DR rulesI · manifest integrity F4/F5 · shares (3σ-binomial, soft)S · KS vs fits, energy balance (soft) E5 · concurrency (warn; --strict-e5 raises)

The manifest then records everything needed to re-create or audit the run: every knob with its source, git SHA + version, per-file row counts and SHA-256s, E5 metrics, noise enforcement stats, realized Dirichlet mixes, and the validation verdict itself.

How the verdict is generated →

E5 report → count active sessions per 15-min tick → max vs charger_count → warn (or raise with --strict-e5)
validation block → run families A–I + soft S/F4/F5 on the written CSVs → passed / errors / warnings into manifest
integrity seals → row counts + SHA-256 per CSV → re-checked by I2/I3 on any later validate

The behavioral coordinate system — real region geometry

consent_default (hand-authored, 5 regions)

Boxes = axes_distribution bounds in the (φ, κ) plane, exactly as configured; % = region weight. A driver is assigned a region ~ Categorical(weights), then (φ, κ, δ) ~ Uniform inside the box.

acn_workplace_baseline (data-anchored, 4+1 regions)

Weights are the empirical user shares from 632 assigned ACN users. Note the different geometry: real workplace drivers plug in rarely (φ ≤ 0.3 dominates: rare_consistent 36% + occasional_consistent 40%); the erratic catch-all fit 0% of this cohort.

The SoC honesty clause. No public charging dataset records state of charge — chargers meter energy, not batteries. Both SoC quantities are therefore modeled priors: arrival SoC is reconstructed by battery_inference from kWh delivered ÷ inferred capacity (capacity inference falls back to class defaults on ~33% of ACN users), and the departure requirement comes from the observed energy request. They are calibrated to be consistent with the ground truth, never claimed to be ground truth. Every fitted leaf carries its provenance (calibration:acn_data_2019_2021_20260628) and fit quality (KS) into the manifest of every run that uses it.
